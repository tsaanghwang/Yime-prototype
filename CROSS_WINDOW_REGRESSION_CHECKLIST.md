# 跨窗口稳定性手工回归清单

这页只保留当前最小但高价值的 Windows 手工回归场景。

目标不是覆盖所有输入法行为，而是尽快确认“候选提交到外部窗口”这条链路没有明显回归。

自动化兜底已补在：

- `yime/input_method/test_app_cross_window_regression.py`

这份清单只负责自动化暂时替代不了的真实窗口现象。

## 回归前准备

1. 启动输入法原型：

```bash
python run_input_method.py
```

1. 打开一个真实外部文本窗口，优先用 Windows 自带记事本。
2. 确认候选框窗口与记事本窗口都可见。
3. 使用当前主线输入方式，确保能正常出候选并完成一次提交。

## 场景 1：首跳焦点不稳时仍能回贴成功

目的：验证第一次前台切换瞬态失败时，二次恢复目标窗口仍能把内容发回外部窗口。

步骤：

1. 在记事本中放置光标。
2. 唤起输入框，输入一段可稳定出候选的编码。
3. 选中一个候选，让系统自动回贴到记事本。
4. 重复 3 到 5 次，过程中有意快速切换注意力，观察是否出现“复制成功但没有真正回贴”的现象。

期望：

1. 候选提交后，目标文本稳定出现在记事本而不是留在剪贴板里未落地。
2. 状态栏出现“已回贴”或“已替换 N 个编码字符”一类成功反馈。
3. 不出现第一下失败、第二下才成功的肉眼可见随机性。
4. 不出现内容误发到输入法自身窗口的情况。

失败信号：

1. 状态栏显示成功，但记事本里没有文本变化。
2. 第一次提交经常无效，第二次才生效。
3. 文本跑进了候选框或别的窗口。

## 场景 2：连续两次跨窗口提交不串长度和目标

目的：验证连续提交时不会把上一次的 `last_replace_length` 或目标窗口状态串到下一次。

步骤：

1. 在记事本中输入一段编码并提交一次候选。
2. 不关闭输入法，立刻再输入第二段编码并再次提交候选。
3. 如果当前模式支持“保留输入框”，再额外做一轮连续提交。
4. 观察每次提交后，记事本中的替换长度、插入位置和候选框焦点是否都正常。

期望：

1. 第二次提交不会删除第一次已经上屏的汉字。
2. 每次替换长度都只对应当前这次输入，不会多删或少删。
3. 提交完成后，焦点行为符合当前模式：该回到候选框时回到候选框，该释放目标时释放目标。
4. 连续两次提交都落到同一个外部目标窗口，不会中途丢目标。

失败信号：

1. 第二次提交把第一次上屏的文本删掉一部分。
2. 明明应该直接插入，却发生了额外 backspace。
3. 第二次提交跑到了错误的窗口。
4. 连续提交后候选框焦点状态异常，必须手工重新点回输入框。

## 建议记录格式

每次手工回归至少记下这几项：

1. 外部窗口类型，例如“记事本”或“VS Code 编辑器”。
2. 是否成功复现问题。
3. 状态栏最后一句话。
4. 是否发生误删、误贴、丢焦点或丢目标窗口。

如果这两条都稳定，再考虑补更细的外部应用兼容清单；
如果这两条都不稳，优先回到
`test_app_cross_window_regression.py` 和 `app_base.py`
的跨窗口提交流程继续收敛。
