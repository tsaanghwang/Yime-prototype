# 码点与中间层策略

## 文档定位

本文档是音元输入系统中关于语义槽位、私用区码点和平台投影的规范性说明。

本文档的目的不是介绍某次临时实现，而是明确以下设计约束：

1. `N01-N24` 与 `M01-M33` 不是可删的临时中间产物，而是系统的语义槽位层。
2. Unicode 私用区字符只是这些语义槽位的码点承载方式，不等于语义本身。
3. 任何测试、重构、数据库调整、键盘布局生成，都不得绕过语义槽位层直接把字符层当作唯一事实源。

如果后续实现与本文档冲突，应优先审查实现，而不是先修改本文档。

## 三层结构

音元输入系统中的编码表示应当稳定地分为三层。

### 1. 语义槽位层

这一层使用槽位编号表达音元身份。

- 噪音槽位：`N01-N24`
- 乐音槽位：`M01-M33`

这一层只回答“它是哪个音元槽位”，不回答“它的 Unicode 码点是什么”。

示例：

- `b -> N01`
- `零首音 -> N22` 或其他约定槽位
- `i1 -> M01 M01 M01`
- `an4 -> M10 M11 M30` 这类三乐音序列

这一层是整个系统的语义真源。键位布局、音节编码、数据库导入、字符投影、排序规则，都应围绕这一层组织。

### 2. 规范码点层

这一层为语义槽位分配一套长期稳定的 Unicode 码点，用于运行时存储、系统内比较、数据库编码、后续理论扩展。

长期建议：

- 规范码点层使用增补私用区字符。
- 在现有历史基础上，优先继续使用 `PUA-B`。

原因：

1. 扩展空间大，适合后续继续加入相关理论字符、辅助标记和派生符号。
2. 全部字符连续规划时更利于分组、排序、逆向还原和长期维护。
3. 与“音元系统作为独立字符系统”的长期目标更一致。

说明：

- `PUA-A` 与 `PUA-B` 在 Unicode 性质上都可行。
- 当前仓库已有较多历史产物使用 `U+100000` 一带，因此继续使用 `PUA-B` 的迁移成本更低。

### 3. 平台投影层

这一层把语义槽位或规范码点再投影到平台更容易处理的一套字符上，用于特定工具链、显示兼容和安装测试。

当前最典型的投影是：

- `BMP PUA` 投影

这一层的用途主要包括：

1. Windows 键盘布局生成和安装链
2. 某些字体、终端、输入框对增补私用区支持不足时的兼容显示
3. 调试阶段更容易目视检查的字符投影

这一层不是语义真源，也不是长期 canonical 层。它是 projection，不是 definition。

## 为什么中间层不能删

`N01-N24` 与 `M01-M33` 的作用不是“多绕一次”，而是把三个容易互相污染的问题分开：

1. 音元语义
2. Unicode 码点选择
3. 键盘与平台兼容

如果删掉语义槽位层，系统会立即出现以下问题：

1. 测试一旦不通，维护者会直接改字符表、改数据库、改布局文件，最后无法判断改动是在修语义、修码点还是修工具链。
2. 一旦从 `PUA-B` 切到 `BMP PUA`，所有依赖字符的文件都会被误认为“语义也变了”。
3. 数据库、运行时编码、键位布局、UI 显示会互相牵连，导致局部调试演变成系统性漂移。

因此，中间层不是冗余层，而是隔离复杂度的必要层。

## 三种字符策略的权衡

### 方案一：全部使用 BMP PUA

优点：

1. Windows 字体、终端、编辑器、SQLite 调试通常更稳。
2. 对当前键盘布局生成、显示排查和兼容测试最省事。
3. 适合作为当前几十个键位字符的统一短期投影。

缺点：

1. 空间有限，后续一旦加入更多理论字符和辅助符号容易拥挤。
2. 更容易与历史私有映射或其他工具链私用映射冲突。
3. 不适合作为长期唯一 canonical 层。

结论：

- 适合作为 projection。
- 不适合作为长期唯一规范码点层。

### 方案二：使用 PUA-A 或 PUA-B

优点：

1. 空间充足，适合完整规划音元系统的字符体系。
2. 更适合做系统内部 canonical 表示。
3. 对未来理论扩展、排序、分组、生成都更干净。

缺点：

1. Windows 某些字体、终端、输入控件、旧工具链支持较差。
2. 某些调试和显示场景不如 BMP 直观。
3. 平台投影和显示兼容工作量更大。

结论：

- 适合作为长期 canonical。
- 需要配套 BMP 投影层。

### 方案三：使用非私用区组合字符

优点：

1. 人工阅读时可能更接近可解释记法。
2. 文档、论文、日志中可作为辅助表示。

缺点：

1. 组合序列在归一化、渲染、比较、索引上不稳定。
2. 视觉接近不等于语义唯一。
3. 不适合作为数据库主表示、键盘布局源或运行时编码。

结论：

- 只能作为辅助可读写法。
- 不能作为主编码层。

## 当前建议

基于仓库现状和后续发展方向，当前建议如下：

1. 保留 `N01-N24` 与 `M01-M33` 作为不可删的语义槽位层。
2. 长期 canonical 继续规划在增补私用区，优先 `PUA-B`。
3. 短期为了 Windows 工具链、键盘布局、输入法实验和显示兼容，可以统一一套 `BMP PUA` 投影。
4. 所有产物文件都应由“语义槽位层 + 码点映射层”生成，而不是手改字符结果。

简化表述如下：

- 语义层：`N/M`
- 规范层：`PUA-B`
- 平台层：`BMP PUA`
- 辅助可读层：非私用区组合字符或拉丁转写

## 文件职责建议

以下职责应当稳定下来。

### 应视为真源或准真源的文件

- `internal_data/manual_key_layout.json`
  - 历史文件名，当前语义应理解为布局真源。
  - 键位与语义槽位的关系。
- `internal_data/key_to_symbol.json`
  - 语义槽位到规范字符的映射
- 后续建议新增：`shouyin_to_symbol_key.json`
  - 首音到 `Nxx` 的语义映射
- 后续建议新增：`ganyin_to_symbol_key_sequence.json`
  - 干音到 `Mxx Mxx Mxx` 的语义映射

### 应视为生成产物的文件

- `yinjie_code.json`
- `yinyuan.klc`
- `syllable/analysis/slice/yinyuan/yinyuan_codepoint.json`
- `syllable/analysis/slice/yinyuan/shouyin_codepoint.json`
- `syllable/analysis/slice/yinyuan/ganyin_to_fixed_length_yinyuan_sequence.json`
- 各类运行时映射、导出 JSON、数据库导入结果

这些文件可以重建，不应手工承担最终真源职责。

## 禁止事项

后续开发和测试中，应避免以下做法：

1. 因测试失败而直接修改数据库中的字符编码，以提高测试通过率。
2. 因平台不兼容而直接跳过 `N/M` 层，把字符表当作唯一事实源。
3. 在没有更新语义映射的情况下，手工批量替换私用区字符。
4. 把某次 BMP 兼容投影误当成整个系统的长期 canonical 层。

## 推荐维护规则

1. 修改语义结构时，先改 `N/M` 层，再重新生成字符层产物。
2. 修改码点区段时，只改码点映射层，不改语义层和布局层。
3. 修改 Windows 键盘布局时，先确认它消费的是语义层还是投影层，不能混用。
4. 对数据库、运行时文件和布局产物，优先重建，不优先手改。

## 最终原则

一句话概括：

`N01-N24` 与 `M01-M33` 是系统的类型系统；`PUA-B` 是长期规范承载；`BMP PUA` 是当前平台投影；非私用区组合字符只用于辅助表达。

只要这个原则不被破坏，后续无论更换码点区、修复工具链，还是重新生成数据库和键盘布局，都不会再把语义、实现和兼容策略混在一起。
