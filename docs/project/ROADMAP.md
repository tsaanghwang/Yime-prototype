# 项目路线图

## 文档定位

本文记录 2026-07 主线闭合后的工程阶段和下一步优先级。当前已经完成到什么程度，以
[当前实现总览](../CURRENT_ARCHITECTURE.md) 为准；候选语料的规模、分层和整理顺序，以
[候选语料库整理路线图](../CANDIDATE_CORPUS_ROADMAP.md) 为准。

路线图不再把已经完成的 N/M 语义真源、三模式生成链、统一来源库或 Windows 前端可消费性验证列为
未来工作，也不把 Python 桌面钩子原型等同于整个 Windows 交付路线。

## 当前阶段

项目当前处于“生产链已闭合，开始整理候选质量并稳定消费者”的阶段。

```text
上游字词与频次来源
  -> source_lexicon.sqlite3（唯一生产真源）
  -> 1732 项有来源的规范带调音节
  -> 正式音节分析与四个 Yinyuan ID
  -> 等长 / 变长 / 省键三模式
  -> 唯一布局投影
  -> Python 原型运行库与 Windows Yime 交接包
  -> Weasel / PIME 等 Windows 消费前端
```

已经落地并受测试或锁检查保护的基础包括：

1. `N01-N24`、`M01-M33` 已由独立语义注册表承载，Yinyuan ID 与字符投影、键位投影分层。
2. `yinjie_code.json`、运行时数据库、resolved layout、KLC 和 crosswalk 已明确为派生或消费产物。
3. 1732 项现行音节采用实例驱动，由正式编码器生成四音元结果；规则缺口会失败并报告。
4. `source_lexicon.sqlite3` 已成为字词、读音、来源证据和频次的唯一生产真源，旧
   `source_pinyin.db` 已退出默认入口和回退链。
5. 等长、变长、省键三模式由同一音元序列确定性派生；键位只从
   `internal_data/manual_key_layout.json` 投影。
6. 原型可以生成 Windows Yime 的唯一等长系统词典输入及拼音审查资产；Windows 侧再确定性派生三模式。
7. Python 桌面输入法原型仍可用于本地交互和回归；Weasel/Rime 与 PIME 已证明能够消费同一数据链。
8. 候选整理覆盖层已经建立，但大规模人工决策、上下文证据和动态组合回放尚未完成。

## 当前最高优先级

### 1. 建立候选质量审查闭环

目标：从“来源与编码正确”推进到“静态候选值得发布”。

近期工作：

- 固定可重复的来源包、拒绝项、多读音和运行候选质量快照；
- 完成 `lexicon_lint` 的只读报告和发版前审阅入口，不让清洗工具直接改生产真源；
- 审查 BCC 频次不低于 10,000 的 517 个未解码字串，保存
  `approved`、`rejected` 或 `deferred` 决策及依据；
- 为高频项提取上下文证据，区分词、专名、固定表达、可动态恢复组合、句法片段和噪声；
- 保持拼音来源合规与候选价值判断分层，不能把“有合法拼音”直接写成“应进入运行候选”。

详细完成标准见 [候选语料库整理路线图](../CANDIDATE_CORPUS_ROADMAP.md)。

### 2. 固定原型到 Windows Yime 的交接协议

目标：让消费者只接收可复核、可重复生成的资产。

近期工作：

- 保持 `yime_full.dict.yaml` 为跨仓库唯一系统词典输入；
- 保持拼音显示、数字标调映射、音节分解和 manifest 的键集及摘要一致；
- 每次来源、编码模式或布局变化后，在 Windows Yime 正式导入器中试派生三模式词典；
- 把原型侧准备、Windows 侧导入和真实用户目录部署分成三个显式步骤；
- 不在 Windows 消费仓库建立第二套拼音到 Yinyuan ID 或键位映射。

交接入口见 [新版词库交接到 Windows Yime](WINDOWS_YIME_LEXICON_HANDOFF.md)。

### 3. 提升真实输入稳定性

目标：分别稳定 Python 原型和系统前端，不混淆两者边界。

近期工作：

- Python 原型继续覆盖组合输入、候选分页、待上屏文本、焦点恢复和跨窗口回贴回归；
- Windows Yime、Weasel 和 PIME 的构建、安装、崩溃及 UI 问题留在各自消费者仓库处理；
- 对三模式切换、候选提交、退格、中英文切换和长时间运行建立可重复烟测；
- 用户词库和学习数据按布局摘要隔离，避免旧布局学习记录污染当前布局。

### 4. 继续清理过时入口和测试假设

目标：让新参与者从入口文档得到当前事实，而不是重走历史分支。

近期工作：

- 删除或明确归档仍把 N/M 真源、统一来源库和系统前端验证写成未来目标的说明；
- 统一 README、FAQ、安装文档和开发文档中的仓库分工；
- 审计直接改 SQLite 行、派生 JSON、布局结果或语义锁摘要的测试；
- 为来源门禁、三模式派生、交接 manifest 和词库质检补充高价值回归。

## 中期目标

### 1. 完成高频候选整理和多读音分层

- 为第二批 BCC 频次 1,000–9,999 的字串形成可解释建议队列；
- 整理高频多读音、词内轻声、专名读音和历史来源冲突；
- 引入 KWIC、邻接频次、左右熵、互信息及保守类别规则；
- 机器或 LLM 只提出可审查建议，不填写无来源拼音或编码。

### 2. 验证动态组合，再精简发布层

- 只使用已批准组件和来源库中已有的合规读音；
- 通过回放比较召回、排序、歧义、内存和速度；
- 仅将达到阈值的条目标记为 `dynamic_recoverable`；
- 在有对照报告前，不整体削减当前静态运行候选。

### 3. 形成可复现的 Windows 试用交付

- 从锁定的原型资产构建版本化交接包；
- 在消费者仓库完成三模式导入、隔离部署和真实应用烟测；
- 明确系统词典、用户词库、学习数据和布局版本的升级/回滚策略；
- 只有在稳定性足够后才制作面向外部试用者的安装包。

## 长期研究与扩展

### 1. 片音语音实验

当前工程只验证符号化音节分析和编码。片音边界、最短听知时限、不同说话人一致性以及插值重建
自然度仍需独立语音实验，不能从四元位置或 Yinyuan ID 反推已经得到实验结论。

### 2. 更强的候选模型

在人工决策集和回放基线形成后，再评估统计模型、语言模型建议、用户自适应排序及领域词库分包。
模型输出必须保留证据和不确定性，不能覆盖来源事实。

### 3. 更多平台

移动端、浏览器端和其他平台只有在语义真源、来源链、候选发布政策与 Windows 消费协议稳定后再评估。
扩展平台应复用同一份语义与布局产物，不建立平行编码。

## 暂不优先

1. 为展示效果另写一套 Web 编码逻辑；
2. 在候选回放前大规模删除静态词条；
3. 让 AI 为未登记拼音猜读音、四音元码或键位；
4. 在消费者仓库修补原型真源问题；
5. 把语音学待验证假设写成已经实现的声学能力。

## 维护原则

1. 当前事实、已完成里程碑和未来目标必须分开。
2. 来源事实、候选决策、音元语义和布局投影必须分层。
3. 新形式没有依据时失败并报告，不以单项手写结果推进进度。
4. 路线图中的动态数字只作快照；长期统计以重新构建后的 manifest 和报告为准。
5. 如果本文与 `CURRENT_ARCHITECTURE.md` 冲突，先核实实现，再立即修正文档。
