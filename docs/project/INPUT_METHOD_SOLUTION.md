# 输入法实现方案

## 文档定位

本文档记录当前仓库内音元输入法原型的真实实现状态、边界和后续方向。

这不是早期的“可行性讨论稿”，而是基于 2026-04-12 当天的修复与重构结果重新整理的实现说明。任何与本文档冲突的旧说明，都应视为过时。

## 当前结论

当前项目已经不再停留在“只能做一个输入框演示”的阶段。

在 Windows + Python 3.12 + pywin32 可用的环境下，当前原型已经具备：

1. 全局低层键盘钩子监听
2. 组合输入时拦截普通码元按键
3. 非激活浮动候选框显示
4. 候选词分页、选择、累积待上屏文本
5. 复制/回贴到外部窗口
6. 关闭时清理钩子并尽量恢复外部窗口交互

因此，当前状态更准确的表述是：

**这是一个 Windows 桌面输入法原型，而不是单纯的候选框演示。**

但它仍然不是系统级 IME，也不是基于 TSF/IMM32 的正式输入法安装包实现。

## 目标流程

当前原型追求的交互流程如下：

```text
用户在任意外部窗口按键
→ 全局键盘钩子接收按键
→ 组合态下拦截码元按键
→ 累积 4 码音元编码
→ 解码候选词
→ 在光标附近显示候选框
→ 用户选字或编辑待上屏文本
→ 回贴到外部窗口
```

这个流程已经能跑通主要链路，但仍属于原型阶段。

## 当前实现架构

### 1. 主应用层

主入口：`python -m yime.input_method.app`

核心文件：

- `run_input_method.py`
- `yime/input_method/app.py`

职责：

1. 初始化解码器、候选框、输入管理器、窗口管理器、剪贴板与键盘模拟器
2. 启动全局键盘监听
3. 将后台线程事件转回 Tk 主线程处理
4. 协调“隐藏待命”和“真正退出”两种行为

### 2. 全局键盘监听层

核心文件：

- `yime/input_method/core/keyboard_listener.py`

当前优先策略：

1. 优先尝试 Win32 低层键盘钩子
2. 失败时再回退到 `pyHook` 或 `pynput`

当前实际状态：

- 在 Python 3.12 环境中，Win32 全局钩子已可正常安装和运行。
- 组合键如 Ctrl、Alt、Win 默认放行，避免破坏系统快捷键。
- 候选框进入手工编辑状态时，按键会让原生输入控件自行处理，不继续强拦截。

### 3. 输入管理层

核心文件：

- `yime/input_method/core/input_manager.py`

职责：

1. 管理组合态缓冲区
2. 处理超时清空、退格、回车、空格、数字选词
3. 驱动候选更新与提交逻辑

当前行为：

- 组合态下可持续累积输入
- 数字键选择当前页候选
- Enter/Space 可提交首选或提交待上屏文本
- 候选不再在核心层截断为前 9 个，而是交给 UI 分页层处理

### 4. 解码层

核心文件：

- `yime/input_method/core/decoders.py`

当前策略：

1. **优先** `yime/pinyin_hanzi.db` → SQLite `runtime_candidates` 视图（默认主路径）
2. SQLite 不可用时，回退到 `.generated/runtime_candidates_by_code_true.json`
3. 运行层仍无候选时，再回退静态层（`pinyin_normalized.json` 解码拼音；可选 `pinyin_hanzi.json` 提供汉字）

当前真实情况：

- 自用只要 `yime/pinyin_hanzi.db` 在且导入链跑通，即可正常查词。
- JSON 导出主要用于人工 diff / 备用，不是日常必需。
- 启动时会打印当前来源，如 `sqlite` 或 `json`。

### 5. 候选框 UI 层

核心文件：

- `yime/input_method/ui/candidate_box.py`

当前特征：

1. 候选框是非激活浮动窗口
2. 支持在需要时切换为可聚焦手工编辑模式
3. 支持分页显示候选词
4. 支持待上屏文本累积、撤销一字、清空待上屏文本
5. 支持从剪贴板读取编码
6. 支持“隐藏到待命小图标”而不是立刻退出

这意味着当前 UI 已不再只是一个静态候选列表，而是一个带编辑能力的辅助输入面板。

## 当前已实现能力

### A. 已经打通的能力

1. **全局监听与组合拦截**
   - 普通码元输入已能进入组合态
   - 非组合型系统快捷键默认放行

2. **浮动候选框显示**
   - 候选框可跟随指针附近出现
   - 支持多显示器和 DPI 缩放场景下的几何修正

3. **候选词解码与回退**
   - 支持运行时候选
   - 支持 SQLite 视图回退
   - 支持静态候选兜底

4. **候选选择与回贴**
   - 选中候选后可复制并回贴到外部窗口
   - 可先累积待上屏文本，再整段提交

5. **手工编辑能力**
   - 输入框可手工粘贴编码
   - 待上屏文本可回退一个字
   - 候选可翻页

6. **退出和隐藏行为**
   - 关闭时会停止键盘监听
   - 会尝试取消调度任务并恢复外部窗口
   - 隐藏时会进入待命图标状态，而不是退出整个程序

### B. 已实现但仍需打磨的能力

1. **自动回贴时机**
   - 当前“单字选择”和“整段上屏”共存，但交互语义仍需继续统一。

2. **手工编辑与全局钩子切换**
   - 当前已做保护，但极端场景仍需继续测试。

3. **候选排序和词频策略**
   - 目前主要依赖数据库视图和已有排序字段。
   - 还不是最终输入法级别的用户自适应排序。

## 当前限制

### 1. 不是系统级 IME

当前实现是“桌面原型 + 全局钩子 + 浮动候选框”，不是注册到 Windows 文本服务框架中的正式 IME。

因此它仍有以下限制：

1. 不具备 TSF/IMM32 级别的系统整合
2. 不保证所有应用都具备一致行为
3. 不具备正式输入法安装包那种系统级稳定性

### 2. 平台限制明显

当前实现几乎完全是 Windows 定向实现。

依赖点包括：

1. Win32 全局键盘钩子
2. 窗口恢复与焦点控制
3. 键盘模拟回贴

### 3. 数据层仍有历史包袱

当前运行时编码和布局链仍存在历史问题：

1. 部分文件仍把字符结果当作事实源
2. `N01-N24` 与 `M01-M33` 的语义层尚未完全回到独立真源文件
3. 数据库与运行时映射仍可能被误当作修复入口

这一点不影响当前原型能跑，但会影响后续长期维护。

## 与旧方案说明的差异

以下判断已经过时，不应继续当作当前状态：

1. “只能用 `pynput`，不能做真正拦截”
2. “当前只能输入框模式，不能在外部应用中直接组合输入”
3. “需要先做全局快捷键唤出才算进入可用状态”

这些说法对应的是更早阶段的实现，不再反映当前仓库状态。

## 当前推荐用法

### 启动方式

```bash
python run_input_method.py
```

或：

```bash
python -m yime.input_method.app
```

### 当前推荐测试流程

1. 在 Windows 环境中使用 Python 3.12 可用环境启动应用
2. 观察启动日志确认键盘钩子是否已安装
3. 在记事本等简单窗口中测试组合输入
4. 测试候选框显示、选字、分页、待上屏文本、隐藏待命和退出恢复

### 当前不建议继续传播的旧描述

1. “只是剪贴板桥接工具”
2. “只能用候选框手动输入”
3. “pywin32 不可用，因此做不了全局输入原型”

## 下一步建议

### 近一步目标

1. 把 `N/M` 语义层重新提升为稳定真源
2. 把运行时字符文件退回生成产物层
3. 继续清理数据库测试与运行时导入边界

### 中期目标

1. 统一候选选择、待上屏文本和回贴的交互语义
2. 继续提升不同应用窗口下的稳定性
3. 明确 canonical 码点层与 BMP projection 的职责分工

### 长期目标

1. 如果继续向正式输入法推进，应评估 TSF/IMM32 级实现
2. 如果主要保留为研究原型，则应优先稳定语义层、生成链和数据链

## 一句话概括

当前实现已经跨过“能不能做出输入法原型”的门槛，进入了“如何把这个 Windows 桌面输入法原型收束成稳定体系”的阶段。
