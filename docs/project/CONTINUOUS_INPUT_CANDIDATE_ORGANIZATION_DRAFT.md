# 连续输入候选组织草案

本文是当前阶段的短技术草案，只回答一个问题：

- 当输入从“单个音节”进入“连续输入/更长上下文”后，候选字词应该如何组织。

本文不讨论：

- 词库净化
- 更细的用户学习策略
- TSF/IMM32 级正式输入法架构

## 设计目标

连续输入阶段希望同时保留三件事：

1. 首音节尚未完整时，仍能给出稳定、可解释的候选反馈
2. 第一个完整音节形成后，能尽快看到高价值词语候选
3. 输入继续增长时，候选组织逐步从“局部桶排序”过渡到“更长上下文词语组织”

当前建议不要把连续输入理解成“把现有排序整体推翻重做”，而应理解成：

- 在现有高频单字兜底和局部词语优先之上，再加一层输入阶段控制。

## 统一命名口径

这几轮 continuous-input 调优，统一建议表述为：

- 短语级连续输入语境调优
- 或更短地说，短语语境调优

这里的“语境”，指的是连续输入过程中可直接观察到的短语级结构信号，例如：

- 当前已经进入哪一层前缀窗口，例如 `6/7/9/10`
- 目标候选是 `2` 字、`3` 字还是 `4` 字
- 多目标之间是否共享稳定词干，例如“毕业/毕业生”“机器/机器人”
- 某些前缀是否只是局部高碰撞噪音，例如“及时/即使”

因此，这套调优不应表述为“语义理解调优”。它并不试图在句子层面理解用户语义，而是在连续输入过程中，用局部短语结构与前缀上下文来更好地组织候选。

## 状态划分

当前建议把连续输入拆成 4 个状态：

### 状态 A：首音节未完成

条件：

- 当前输入还不能按运行时单音节码表解析出完整音节

目标：

- 告诉用户当前正在输入第一个音节
- 候选以“可兜底的单字/前缀提示”为主，不把它误包装成词语精确命中

### 状态 B：首音节刚完成

条件：

- 当前输入刚好解析出 1 个完整音节，且没有后续残码

目标：

- 保留高频单字兜底
- 同时启动“候选词语优先”，让高碰撞单字码尽快露出常用词

这是当前局部词语优先规则主要生效的阶段。

### 状态 C：连续输入进行中

条件：

- 已完成至少 1 个完整音节
- 当前还存在未完成的后续音节残码

目标：

- 不只看最后一个音节
- 用“最近若干完整音节 + 当前残码前缀”组织词语候选
- 让候选逐步反映更长上下文，而不是每次都退回单音节思路

当前实现口径：

- 最近 `1` 个完整音节 + 当前残码，对应 `long-context-prefix-1`，候选上限 `32`
- 最近 `2` 个完整音节 + 当前残码，对应 `long-context-prefix-2`，候选上限 `24`
- 最近 `3` 个完整音节 + 当前残码，对应 `long-context-prefix-3`，候选上限 `16`

这样做的目的，是让上下文越长时，前缀池越收窄，避免连续输入一拉长就重新放大高碰撞前缀桶。

### 状态 D：连续词语整码命中

条件：

- 当前输入由多个完整音节组成
- 没有残码

目标：

- 优先按完整多音节词语命中组织候选
- 不再把它当成单音节前缀扩展场景

当前实现里，最多保留最近 4 个完整音节作为整词查找窗口。

## 每个阶段查什么候选

建议按以下结构组织：

- A 首音节未完成
  - 主要查找对象：单字前缀候选 + 等待提示
  - 备注：重点是可解释反馈，不是假装精确命中
- B 首音节刚完成
  - 主要查找对象：单字精确桶 + 单音节词语前缀扩展
  - 备注：当前“高频单字兜底 + 候选词语优先”主要叠加在这里
- C 连续输入进行中
  - 主要查找对象：最近 `1/2/3` 个完整音节 + 当前残码的词语前缀候选
  - 备注：当前口径分别限制为 `32/24/16`，重点从单字桶过渡到连续词语组织
- D 连续词语整码命中
  - 主要查找对象：多音节完整词语候选
  - 备注：优先完整词命中，其次才考虑局部补充

## 现有两层策略怎么叠加

### 高频单字兜底

作用：

- 解决“词语候选还不稳定时，至少先有常用字能落地”的问题

建议作用阶段：

- 状态 A：主用
- 状态 B：保留
- 状态 C：降级为兜底
- 状态 D：只在完整词语候选不足时兜底

### 候选词语优先

作用：

- 解决高碰撞单字桶下，首屏全被单字占据的问题

建议作用阶段：

- 状态 B：主用
- 状态 C：可延伸，但不应继续只看单音节高碰撞桶
- 状态 D：让位给完整多音节词语命中

换句话说：

- 高频单字兜底负责“没有词时仍可输入”
- 候选词语优先负责“第一个音节完成后尽快见词”
- 连续输入组织负责“输入继续增长后不要退回局部短视排序”

## 当前实现与本草案的关系

当前实现已经具备这份草案的雏形：

1. 首音节未完成时，会显示“当前首音节未完成，已输入 x 码”
2. 第一个完整音节形成后，会进入单音节精确桶 + 词语前缀扩展
3. 多音节残码时，会按“最近完整音节 + 当前残码”继续组织候选
4. 多音节整码时，会进入完整词语模式
5. 仓库已增加独立的 `internal_data/continuous_input_priority_rules.json` 入口，用于试点状态 C/D 的更长上下文人工规则

本轮 continuous-input 调优到当前为止，可以先按“短语级连续输入语境调优”的试点阶段收束：

- 单字兜底、首音节词语优先、连续输入短语出屏，已经形成可并存的三层组织关系
- 状态 C 的 `long-context-prefix-1/2/3` 已固定为 `32/24/16` 的候选上限口径
- continuous 规则生成已按词长分层放宽：`2` 字目标优先看 `6/7`，
  `3` 字目标扩到 `6/7 + 9/10`，`4` 字目标后移到 `9/10 + 13/14`
- `6/7` 多目标规则已做噪音过滤：保留共享稳定词干的组，
  以及少量显式允许的变体组；`及时/即使`、`灵魂/灵活`
  这类噪音前缀不再保留
- 当前生成结果里，剩余多目标 continuous 组已经只落在 `6/7`；`9/10+` 位置不再保留多目标组，可视为这一轮收窄后的维护边界
- 保留下来的代表组，例如“其他/其它”“毕业/毕业生”“机器/机器人”
  “计算/计算机”“验证/验证集/验证码”，已经有回归测试锁定其
  “稳定在第一屏前排”的行为

因此，这一轮更适合视为“已完成最小验收并进入维护态”，而不是继续在同一层面无限展开。后续若没有新的真实样本回归，continuous-input 主线可以先收口。

## 下一步建议

如果后续还要继续推进，建议顺序是：

1. 继续只在真实样本触发回归时补测试或小修规则，不再盲目扩大 continuous rule 覆盖面
2. 再根据实际输入观察，决定是否需要把“第一屏前排”细化成更明确的位置目标，例如前 `2/3/5`
3. 将主要精力转到下一类问题，例如专用字、罕用字和扩展字符的可达性与兜底体验

这样可以避免 continuous-input 这一轮已经收束的策略再次失焦，也能把新一轮工作切到更值得投入的问题域。
