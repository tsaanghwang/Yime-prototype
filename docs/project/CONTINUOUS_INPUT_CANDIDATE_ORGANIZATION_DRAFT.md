# 连续输入候选组织草案

本文是当前阶段的短技术草案，只回答一个问题：

- 当输入从“单个音节”进入“连续输入/更长上下文”后，候选字词应该如何组织。

本文不讨论：

- 词库净化
- 更细的用户学习策略
- TSF/IMM32 级正式输入法架构

## 设计目标

连续输入阶段希望同时保留三件事：

1. 不足 4 码时，仍能给出稳定、可解释的候选反馈
2. 打满第一个音节后，能尽快看到高价值词语候选
3. 输入继续增长时，候选组织逐步从“局部桶排序”过渡到“更长上下文词语组织”

当前建议不要把连续输入理解成“把现有排序整体推翻重做”，而应理解成：

- 在现有高频单字兜底和局部词语优先之上，再加一层输入阶段控制。

## 状态划分

当前建议把连续输入拆成 4 个状态：

### 状态 A：首音节未完成

条件：

- 总输入长度 `1-3` 码

目标：

- 告诉用户当前正在输入第一个音节
- 候选以“可兜底的单字/前缀提示”为主，不把它误包装成词语精确命中

### 状态 B：首音节刚完成

条件：

- 总输入长度 `4` 码

目标：

- 保留高频单字兜底
- 同时启动“候选词语优先”，让高碰撞单字码尽快露出常用词

这是当前局部词语优先规则主要生效的阶段。

### 状态 C：连续输入进行中

条件：

- 已完成至少 1 个完整音节
- 当前还存在未完成的后续音节，也就是 `4n + 1` 到 `4n + 3` 码

目标：

- 不只看最后一个音节
- 用“最近若干完整音节 + 当前残码前缀”组织词语候选
- 让候选逐步反映更长上下文，而不是每次都退回单音节思路

### 状态 D：连续词语整码命中

条件：

- 当前输入由多个完整音节组成
- 没有残码，也就是 `8/12/16...` 码整

目标：

- 优先按完整多音节词语命中组织候选
- 不再把它当成单音节前缀扩展场景

当前实现里，最多保留最近 4 个完整音节作为整词查找窗口。

## 每个阶段查什么候选

建议按下表组织：

| 状态 | 主要查找对象 | 备注 |
| --- | --- | --- |
| A 首音节未完成 | 单字前缀候选 + 等待提示 | 重点是可解释反馈，不是假装精确命中 |
| B 首音节刚完成 | 单字精确桶 + 单音节词语前缀扩展 | 当前“高频单字兜底 + 候选词语优先”主要叠加在这里 |
| C 连续输入进行中 | 最近若干完整音节 + 当前残码的词语前缀候选 | 重点从单字桶过渡到连续词语组织 |
| D 连续词语整码命中 | 多音节完整词语候选 | 优先完整词命中，其次才考虑局部补充 |

## 现有两层策略怎么叠加

### 高频单字兜底

作用：

- 解决“词语候选还不稳定时，至少先有常用字能落地”的问题

建议作用阶段：

- 状态 A：主用
- 状态 B：保留
- 状态 C：降级为兜底
- 状态 D：只在完整词语候选不足时兜底

### 候选词语优先

作用：

- 解决高碰撞单字桶下，首屏全被单字占据的问题

建议作用阶段：

- 状态 B：主用
- 状态 C：可延伸，但不应继续只看单音节高碰撞桶
- 状态 D：让位给完整多音节词语命中

换句话说：

- 高频单字兜底负责“没有词时仍可输入”
- 候选词语优先负责“第一个音节完成后尽快见词”
- 连续输入组织负责“输入继续增长后不要退回局部短视排序”

## 当前实现与本草案的关系

当前实现已经具备这份草案的雏形：

1. 不足 4 码时，会显示 `当前 x/4 码，继续输入`
2. 打满 4 码时，会进入单音节精确桶 + 词语前缀扩展
3. 多音节残码时，会按“最近完整音节 + 当前残码”继续组织候选
4. 多音节整码时，会进入完整词语模式
5. 仓库已增加独立的 `internal_data/continuous_input_priority_rules.json` 入口，用于试点状态 C/D 的更长上下文人工规则

但当前还没有把这 4 个阶段作为一套明确的候选组织策略单独收束出来，这正是下一阶段设计工作的目标。

## 下一步建议

如果按最小增量推进，建议顺序是：

1. 先把这 4 个状态固化成测试场景
2. 再明确每个状态的候选混排比例与回退条件
3. 最后再考虑是否引入更长上下文的人工规则或用户学习

这样可以避免在连续输入阶段一开始就把排序、词库、学习策略同时搅在一起。
