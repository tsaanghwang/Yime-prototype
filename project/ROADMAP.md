# 项目路线图

## 文档定位

本文档描述当前仓库在 2026-04-12 之后的实际推进方向。

它不再以前期的 Web 原型、多平台展示或历史实验分支为中心，而是以当前已经跑通的 Windows 桌面输入法原型为基线，定义接下来该优先完成什么。

其中涉及理论扩展、更多输入模式或更远期能力的部分，应优先理解为研究方向和保留目标，不等于当前阶段全部进入实现排期。

如需理解当前已经做到哪里，请先看：

- `INPUT_METHOD_SOLUTION.md`
- `../CODEPOINT_POLICY.md`
- `../SOURCE_AND_ARTIFACTS.md`

## 当前基线

当前项目已经具备以下现实基础：

1. Windows + Python 3.12 + `pywin32` 的桌面输入法原型可启动
2. 全局低层键盘钩子已打通主要链路
3. 候选框、分页、待上屏文本、复制/回贴已可用
4. 运行时候选已具备 SQLite 回退链路
5. 文档主入口已基本收拢到当前主线

因此，下一阶段的重点不应再是“再写一个 Web 演示”，而应是：

- 稳定当前 Windows 原型
- 把数据链和语义层拆回正确结构
- 清理测试和文档中的历史误导

## 近期目标（2026 Q2）

### 1. 稳定当前 Windows 输入法原型

目标：把已经跑通的原型从“能演示”推进到“可持续迭代”。

重点包括：

- 继续验证全局钩子、候选框、回贴在常见 Windows 应用中的行为
- 收敛“单字选择”和“整段待上屏文本提交”的交互语义
- 继续打磨手工编辑模式与全局拦截模式之间的切换边界
- 降低焦点恢复、窗口切换、回贴时机带来的不稳定感
- 建立可重复生成的效率基线报告，先用现有候选导出数据追踪词语重码、单字重码和首屏容纳率

### 2. 把 `N/M` 语义层重新提升为真源

目标：修复当前最关键的长期维护问题。

重点包括：

- 明确 `N01-N24` 与 `M01-M33` 是不可删的 Yinyuan ID 层
- 新增独立的首音/干音语义真源文件
- 停止把字符结果文件或数据库当作最终事实源
- 让码点层与语义层重新解耦

### 3. 让运行时字符文件退回生成产物层

目标：把当前混乱的“真源/产物”边界重新建立起来。

重点包括：

- 明确 `yinjie_code.json`、各类 `*_codepoint.json`、`yinyuan.klc` 都是产物
- 从语义真源层自动生成运行时字符文件
- 避免测试、修复或实验过程直接手改运行时映射结果

### 4. 清理测试体系中的错误假设

目标：避免再次出现“为了让测试通过而改错层级”的情况。

重点包括：

- 审计会写数据库或改产物文件的测试
- 限制测试直接改数据库字符编码、布局产物或运行时映射
- 让测试围绕语义层、生成链和可验证行为组织，而不是围绕偶然产物组织
- 补齐当前 Windows 输入法原型的关键回归测试

### 5. 完成文档收口

目标：让仓库只保留当前真实有效的入口文档。

重点包括：

- 继续清理旧的状态报告、旧安装说明、旧使用说明和阶段性方案稿
- 把 README、FAQ、USAGE、INSTALL 等入口文档统一到当前主线
- 明确哪些文档是规范性文档，哪些只是审计或过渡文档

## 中期目标（2026 Q3-Q4）

### 1. 完成稳定的生成链

目标：让布局、码点、音节编码和数据库导入都由可重建流程驱动。

重点包括：

- 从语义层生成 canonical 层和 projection 层
- 统一生成 `yinjie_code.json`、布局产物和数据库导入资产
- 降低手工维护字符映射文件的需求

### 2. 提升输入法级别的交互质量

目标：把当前原型从“能工作”提升到“更接近正式输入法体验”。

重点包括：

- 优化候选排序和词频策略
- 在已有内部效率基线稳定后，引入外部输入法对照和真实录入实验
- 设计更稳定的用户词库与频次更新机制
- 改善分页、待上屏文本、撤销与整段提交体验
- 补齐更多常见应用场景下的兼容性测试

### 3. 建立更稳的工程基线

目标：让项目后续迭代不再依赖临时人工判断。

重点包括：

- 收敛持续集成与自动测试
- 补齐关键脚本和生成链的开发文档
- 为高风险区域增加审计脚本和一致性报告

## 长期目标

### 1. 评估系统级 IME 路线

当前实现仍然不是 TSF / IMM32 系统级输入法。

长期可以评估：

- 是否要进入正式 IME 架构
- 是否要生成系统安装包
- 是否要从桌面原型升级为正式 Windows 输入法实现

### 2. 建立稳定的字符系统与工具链生态

长期目标不是只让某一版布局能工作，而是让以下结构稳定下来：

- 语义层
- 规范码点层
- 平台投影层
- 可重建的运行时产物层

### 3. 再决定是否扩展到更多端

跨平台、移动端、浏览器端或其他界面形态不应先于核心数据链与 Windows 原型稳定化。

只有在以下前提满足后，才适合继续讨论：

- 语义层稳定
- 生成链稳定
- 当前 Windows 原型行为稳定

## 暂不优先的方向

以下方向不是永久放弃，但在当前阶段不应排在前面：

1. 以 Web 界面为中心的 UI 美化
2. 以 React/Node 为中心的前端路线扩张
3. 跨平台展示型原型优先于当前 Windows 主线
4. 在数据链未稳定前就大量扩展高级功能点

## 维护原则

维护本路线图时，遵循以下原则：

1. 先写真实基线，再写未来目标，不把历史愿望当当前状态。
2. 先修结构性问题，再扩功能，不用表层功能掩盖底层混乱。
3. 先稳语义层与生成链，再谈跨平台或系统级扩张。
4. 如果路线图与 `INPUT_METHOD_SOLUTION.md` 冲突，以真实实现状态为准，优先修路线图。

> 本路线图应随实现状态持续更新，但更新时应优先反映真实阶段，不重复保留过时方向。
